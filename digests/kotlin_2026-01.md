### [Jetpack Compose Memory Leaks: A Reference-Graph Deep Dive](https://medium.com/@mohansankaran/jetpack-compose-memory-leaks-reference-graph-deep-dive)

Compose doesn't leak by default — most Compose memory leaks are plain Kotlin reference leaks where a long-lived object (ViewModel, singleton, app-scoped coroutine) holds a reference to something composition-scoped (Activity Context, composable lambda, remembered object). The key mental model: leaks happen when composition-scoped references escape into longer-lived holders. The article walks through reference graph analysis with LeakCanary and practical patterns for ensuring composition-scoped objects don't outlive their intended scope.

### [Building AI Agents in Kotlin – Part 4: Delegation and Sub-Agents](https://blog.jetbrains.com/ai/2026/01/building-ai-agents-in-kotlin-part-4/)

Part 4 of the Koog series covers delegation and sub-agents for structuring scalable agent architectures. After adding tracing in Part 3, delegation allows an orchestrator agent to hand off specialised tasks to sub-agents with narrower tool access — improving reliability and cost. The pattern maps naturally to Kotlin's coroutine model: sub-agents run in child scopes and results propagate back to the parent. Useful for multi-step workflows where a single agent context becomes too broad.

### [Compose Multiplatform 1.10.0: Navigation 3, Stable Hot Reload & More](https://blog.jetbrains.com/kotlin/2026/01/compose-multiplatform-1-10-0/)

### [The Journey to Compose Hot Reload 1.0](https://blog.jetbrains.com/kotlin/2026/01/compose-hot-reload-1-0/)

Compose Hot Reload 1.0 reaches stable, bundled with CMP 1.10+. The engineering challenge was enabling class redefinition at runtime without restarting the app — this required Kotlin compiler changes to make composable functions redefinable via JVMTI. The 1.0 release is zero-configuration: click "Run with Hot Reload" in IntelliJ IDEA for any JVM/Desktop module. Mobile support (Android/iOS) remains in progress. The post covers the compiler architecture changes required to make this work safely.

### [Kotlin DSLs in 2026: Patterns That Stood the Test of Time](https://dev.to/eugenep/kotlin-dsls-in-2026)

Eight years after Kotlin DSLs were first popularized, extension functions, lambdas with receivers, and operator overloading remain the core building blocks — and have become standard practice in production JVM codebases. The post revisits the original Java builder problem, shows how the same DSL techniques underpin Gradle Kotlin DSL, Ktor routing, and Jetpack Compose itself, and argues that these patterns work because they're grounded in Kotlin's type system rather than being syntactic sugar. A useful retrospective for developers who want to understand why DSL APIs are designed the way they are.

### [Kotlin 2.3.0 Released](https://blog.jetbrains.com/kotlin/2026/01/kotlin-2-3-0-released/)

Kotlin 2.3.0 ships new language features including: unused return value checker (opt-in warning when you ignore the return value of a function annotated `@CheckReturnValue`), explicit backing fields (declare `field` directly in a property block), and changes to context-sensitive resolution. Kotlin/JVM adds Java 25 support. Kotlin/Native improves interop. The release also promotes several previously experimental features to stable. Use `languageVersion = "2.3"` in your Gradle config to enable new features.

### [Don't Block Suspend Functions](https://publicobject.com/2026/01/22/dont-block-suspend-functions/)

Jesse Wilson demonstrates concretely why calling blocking code inside a `suspend` function breaks coroutines: if all coroutine dispatcher threads are blocked, cooperative cancellation stops working and the entire coroutine machinery stalls. The fix is always `withContext(Dispatchers.IO)` to offload blocking work to the IO thread pool, which is unbounded. The article uses a clear test case showing the failure mode — useful to share with teams that are migrating from callbacks or threads to coroutines and haven't fully internalized the cooperative model.

### [How to Avoid Common Pitfalls With JPA and Kotlin](https://blog.jetbrains.com/idea/2026/01/jpa-kotlin-pitfalls/)

JPA entities must be `open` (non-final) for lazy loading to work via proxies, but Kotlin classes are `final` by default — requiring either the `kotlin-allopen` plugin or manual `open` annotations. Other pitfalls: `data class` identity semantics conflict with JPA's entity identity model, and Kotlin's non-null types create friction with JPA's nullable-by-default column conventions. The post recommends modeling JPA entities as open mutable classes using the `@Entity` + `kotlin-allopen` combination and avoiding `data class` for entities.

### [Shared Internals: Kotlin's New Proposal for Cross-Module Visibility](https://jaewoong.dev/shared-internals-kotlin/)

The `internal` modifier hides implementation details within a module, but modularized codebases often need to share internals across related modules (e.g., test modules accessing production internals, or coroutines sharing implementation across artifacts). The "shared internal" KEEP proposes a transitive, opt-in cross-module visibility level — a module explicitly declares which other modules can access its internals. This would eliminate the current workarounds of either making things `public` or duplicating code. Still a proposal; worth tracking if you maintain multi-module libraries.
