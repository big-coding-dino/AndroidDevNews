### [Increase Guidance and Control over Agent Mode with Android Studio Panda 3](http://android-developers.googleblog.com/2026/04/Increase-Guidance-and-Control-over-Agent-Mode-with-Android-Studio-Panda-3.html)
Android Studio Panda 3 is now stable, delivering two meaningful upgrades to Agent Mode: **agent skills** and **granular agent permissions**. **Agent skills** let you create `.md` files inside a `.skills/` directory at your project root that teach the agent your team's conventions and workflows. Each skill file has a name and description, and can include scripts, assets, and references. The agent picks up skills automatically or you can invoke them with `@skill-name`. Use cases include custom code review rules tied to your org's standards, or onboarding the agent to an internal library it wouldn't otherwise know about. **Granular agent permissions** give you explicit control over what Agent Mode can do — read files, run shell commands, access the web. Each category can be individually approved with an "Always Allow" rule for trusted operations, so routine tasks don't interrupt your flow. Sensitive files (SSH keys, etc.) always require manual sign-off. There's also an optional **sandbox mode** for stricter, isolated execution when you need it. Panda 3 also ships an **Empty Car App Library App** template that generates a working scaffold for both Android Auto and Android Automotive OS, eliminating the boilerplate setup that previously made car app development cumbersome to start.

**Why it matters:** Agent skills directly address one of the core friction points with AI-assisted development — the agent not knowing your codebase's conventions. Instead of front-loading every prompt with context, you encode that knowledge once and reuse it across your team. The permissions model tackles the opposite problem: agents that are too autonomous. The "Always Allow" + sandbox combination is a practical middle ground between blocking the agent at every step and giving it unchecked access, which matters more as agents run longer, multi-step tasks where a mistake is expensive to undo.

### [Announcing Gemma 4 in the AICore Developer Preview](http://android-developers.googleblog.com/2026/04/AI-Core-Developer-Preview.html)
Google's **AICore Developer Preview** now includes **Gemma 4**, the new open model that forms the foundation of the upcoming **Gemini Nano 4**. Code written against Gemma 4 today will run on Gemini Nano 4 devices when they ship later this year — no migration required. Gemma 4 on Android ships in two sizes optimized for different trade-offs: **E2B** (fast) and **E4B** (full). Both support **140+ languages** natively and add **multimodal input** (text, image, audio). Compared to previous versions, the new model is up to **4x faster** and uses up to **60% less battery**. Integration uses the **ML Kit Prompt API**. A new `modelConfig` parameter on `GenerationConfig` lets you explicitly target E2B or E4B via `ModelReleaseTrack.PREVIEW` and `ModelPreference.FAST`/`ModelPreference.FULL`. You check availability with `GenerativeModel.checkStatus()` before running inference. You can prototype prompts directly in **Android Studio** before committing to code. Models run on AI accelerators from Google, MediaTek, and Qualcomm on AICore-enabled devices; non-AICore devices fall back to CPU (not production-representative). The **AI Edge Gallery** app is available for testing on unsupported hardware. Planned additions during the preview period include **tool calling**, **structured output**, **system prompts**, and **thinking mode** in the Prompt API.

**Why it matters:** On-device inference with a multimodal, multilingual model at this performance level removes a major blocker for shipping AI features without a server dependency. The Gemma 4 → Gemini Nano 4 upgrade path means preview work isn't throwaway code — you're building against the production API surface now. The explicit model variant selection (`E2B`/`E4B`) also gives developers real control over the latency/quality trade-off, which matters in resource-constrained mobile environments.

### [Android Studio supports Gemma 4: our most capable local model for agentic coding](http://android-developers.googleblog.com/2026/04/android-studio-supports-gemma-4-local.html)
Android Studio now supports **Gemma 4** as a local AI model for coding assistance, continuing the model-choice flexibility introduced in January. Gemma 4 is a **Mixture-of-Experts (MoE)** model trained specifically on Android development that runs entirely on your local machine via **LM Studio** or **Ollama** — no API key or internet connection required for core operations. The model integrates with **Agent Mode** and is capable of multi-file, multi-step agentic tasks: generating new features (writing Kotlin + Jetpack Compose by default), refactoring across files (e.g., migrating hardcoded strings to `strings.xml`), and iteratively fixing build errors and lint issues. All processing happens locally, using your machine's GPU and RAM.

Three model sizes are available with the following hardware requirements (RAM figures include Android Studio overhead):

| Model | RAM | Storage |
|---|---|---|
| Gemma E2B | 8 GB | 2 GB |
| Gemma E4B | 12 GB | 4 GB |
| Gemma 26B MoE | 24 GB | 17 GB |

The 26B MoE variant is the recommended choice for most developers. Setup involves installing LM Studio or Ollama, adding it under **Settings > Tools > AI > Model Providers**, downloading the model, then selecting Gemma 4 in Agent Mode.

**Why it matters:** Local LLMs for agentic coding have historically lagged behind cloud models in reasoning quality. Gemma 4 is the first Android-specific model trained for tool calling and multi-step agentic workflows, which closes that gap meaningfully. For developers in regulated industries or corporate environments with strict data policies, this makes Agent Mode actually usable — your code never leaves the machine. It also removes quota constraints for heavy users, making iterative agentic workflows (build-fix loops, large-scale refactors) practically free to run beyond the initial hardware cost.

### [Android Studio supports Gemma 4: our most capable local model for agentic coding](https://android-developers.googleblog.com/2026/04/android-studio-supports-gemma-4-local.html)
Android Studio now supports **Gemma 4** as a local AI model for coding assistance, including full **Agent Mode** support. Gemma 4 was specifically trained on Android development data and features agentic tool-calling capabilities — meaning it can take multi-step actions across your codebase, not just generate snippets. The model runs entirely on your machine via a local LLM provider like **Ollama** or **LM Studio**. Setup involves adding the provider under `Settings > Tools > AI > Model Providers`, downloading the model, then selecting it in Agent Mode. Three model sizes are available: **Gemma E2B** (8GB RAM, 2GB storage), **Gemma E4B** (12GB RAM, 4GB storage), and the recommended **Gemma 26B MoE** (24GB RAM, 17GB storage). In Agent Mode, Gemma 4 can handle real agentic tasks: scaffolding new features using Kotlin and Jetpack Compose, bulk refactors like migrating hardcoded strings to `strings.xml` across multiple files, and iterative build-fix loops where it reads compiler output and applies fixes until the build passes.

**Why it matters:** Running AI agents locally removes two significant blockers: API quota limits and code privacy concerns. For developers at companies with strict data handling policies, sending source code to a remote model is often a non-starter. Gemma 4 lets those developers access the same agentic workflows — multi-file edits, build error resolution, feature scaffolding — without any code leaving the machine. The 26B MoE architecture also signals that local model quality is closing the gap with hosted alternatives, making the privacy/cost tradeoff increasingly viable for day-to-day use.

### [Agentic Interaction Using AppFunctions](https://dev.to/tkuenneth/agentic-interaction-using-appfunctions-m8k)
**AppFunctions** is Google's latest API for exposing app capabilities to on-device agents like Gemini. It replaces a graveyard of prior attempts — App Actions (too rigid, predefined intents only), Android Slices (deprecated 2021, too much UI maintenance), Direct Actions (required foreground app), and the Assist API (screen scraping without reliable parsing) — each of which failed because they forced apps into static shapes that dynamic intelligence couldn't work with. AppFunctions flips the model: instead of deep-linking into a screen, agents call discrete, self-describing functions your app publishes. The `androidx.appfunctions` Jetpack library (`1.0.0-alpha08`) began alpha rollout in May 2025. Platform support (`android.app.appfunctions`) landed in API 36 (Android 16). The minimum viable Gradle setup requires `compileSdk = 36`, KSP, three Jetpack coordinates (`appfunctions`, `appfunctions-service`, `appfunctions-compiler`), and a `ksp { arg("appfunctions:aggregateAppFunctions", "true") }` block. A non-obvious Gradle gotcha: you must manually wire each `merge*Assets` task to depend on its matching `ksp*Kotlin` task, or the generated `app_functions.xml` / `app_functions_v2.xml` assets won't be packaged in time. Declaring a function is straightforward. Annotate a `suspend fun` with `@AppFunction(isDescribedByKDoc = true)`, accept `AppFunctionContext` as the first parameter, and annotate any custom data classes with `@AppFunctionSerializable`. KDoc on the method feeds directly into the metadata agents consume — parameter semantics included. The `appfunctions-service` dependency auto-merges a `PlatformAppFunctionService` into your manifest via manifest merging, so there's no manual service registration. On-device discovery is backed by **AppSearch**, handled entirely by the library. Validation is done via adb: `adb shell cmd app_function list-app-functions | grep -F <your.package>` confirms the OS indexer picked up your functions. Execution can be driven from the shell too, passing parameters as AppSearch-style JSON arrays (`"app1":["com.foo"]`). The `app_function` shell path requires API 36.1+ AVD images — earlier 36.0 images may return `No shell command implementation`. For automated testing, `AppFunctionTestRule` covers local JVM/Robolectric runs; instrumented tests on an API 36+ image cover the full AppSearch sync and shell path. Two production caveats worth noting: with R8/minification enabled, the Jetpack artifacts ship consumer ProGuard rules that handle most of the generated surface, but you should smoke-test a release build and add targeted `-keep` rules for your own `@AppFunction` classes if discovery breaks after shrinking. On the trust model, only privileged system assistants hold the `EXECUTE_APP_FUNCTIONS` permission — ordinary apps can't invoke AppFunctions from the Play Store. Still, treat incoming parameters as untrusted input. Google describes the end-to-end agent path as experimental and capacity-limited as of early 2026, with Android 17 slated to broaden reach. UI automation (for flows without structured AppFunctions) is previewing on Galaxy S26 and select Pixel 10 hardware.

**Why it matters:** AppFunctions is the first Android agent integration API with a realistic path to mainstream adoption — it's Jetpack-backed, KSP-driven, and doesn't require your app to be in the foreground or your UI to be screen-scraped. For developers, the surface area is small (one annotation, one Gradle flag, one context parameter), but the payoff is that Gemini and future agents can call your app's capabilities directly as tools. Getting the Gradle wiring right today, and validating packaging with the adb checks, puts your app in position before the API exits beta and Google widens assistant access.
