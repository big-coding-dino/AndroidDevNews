### [How LLMs Work – The 20-Minute Explainer (Fragmented Ep. 303)](https://fragmentedpodcast.com/episodes/303/)

Kaushik and Iury walk through the full LLM pipeline — tokenization (text → tokens), embeddings (tokens → high-dimensional vectors encoding semantic meaning), and inference (next-token prediction via attention) — in a format concise enough to actually retain. The episode uses colour space (RGB) as an analogy for embedding dimensions, which makes the abstract geometry more concrete. Useful for Android developers who use AI tools daily but have never built a clear mental model of what's actually happening under the hood. *(Summary-only — full transcript not available.)*

### [Koog x ACP: Connect an Agent to Your IDE and More](https://blog.jetbrains.com/ai/2026/02/koog-x-acp-connect-an-agent-to-your-ide-and-more/)

The **Agent Client Protocol (ACP)** is an open protocol (with SDKs for Kotlin, Python, TypeScript, and Rust) that defines how agents communicate with clients — notifying the client of LLM responses, tool calls, and requesting file-edit permissions. Koog now has **native ACP support**, meaning a Koog agent can be made IDE-connectable with minimal glue code and will appear directly in the IntelliJ AI Chat panel. The post explains what ACP-compliance requires at the Koog API level and how to wire a custom Koog agent into IntelliJ. This is the missing piece from earlier ACP tutorials — those showed the client side; this shows how to make your agent a compliant server.

### [The First Beta of Android 17](https://android-developers.googleblog.com/2026/02/the-first-beta-of-android-17.html)

Android 17 Beta 1 (API level 37) lands with a significant mandatory change: **apps targeting SDK 37 can no longer restrict orientation or resizability on large screens (sw > 600dp)**. The developer opt-out that was technically available in Android 16 is removed entirely. Additional focus areas include camera/media enhancements, connectivity tooling, and expanded companion device profiles. The release also marks the end of the traditional "Developer Preview" model — Android development now flows through a continuous **Canary channel**, providing a smoother, always-on stream of builds rather than discrete preview drops.

### [The Next Software Crisis Won't Be About Writing Code](https://blog.kotzilla.io/the-next-software-crisis-wont-be-about-writing-code)

Arnaud Giuliani (Kotzilla) argues that AI making code production cheap doesn't eliminate technical debt — it accelerates it. The historical parallel: the UML/MDA era promised model-driven code generation would solve software complexity; it didn't, because generating code doesn't generate understanding of the problem. AI is in the same position. The risk isn't that AI writes bad code; it's that developers ship AI-generated code they don't understand well enough to maintain, review, or debug. The prescription: invest in architectural thinking and clear problem modelling *before* prompting, not after. A JetBrains survey cited in the article found 22.8% of Kotlin developers already feel growing pains from expanding codebases.

### [AVD Buddy](https://github.com/alexstyl/avdbuddy)

A native macOS menu-bar app for managing Android Virtual Devices without opening Android Studio. Supports browsing, launching (double-click), creating, duplicating, renaming, and deleting AVDs, plus downloading system images from Google. Covers phones, tablets, foldables, TVs, and Wear OS emulators. Reads the SDK from `ANDROID_SDK_ROOT` / `ANDROID_HOME` / `~/Library/Android/sdk` and AVDs from `~/.android/avd`. Install via DMG from GitHub Releases. Built with Xcode; requires `avdmanager`, `sdkmanager`, `adb`, and the Android Emulator.

### [The Intelligent OS: Making AI Agents More Helpful for Android Apps](https://android-developers.googleblog.com/2026/02/the-intelligent-os-making-ai-agents.html)

Google introduces **AppFunctions** — a Jetpack library and platform API that lets apps expose self-describing functions discoverable and invocable by AI agents (like Gemini) via natural language, without the agent needing to drive the UI. This is the Android counterpart to MCP server exposure: your app registers functions with metadata, and agents can find and call them. A companion **AI-driven UI automation framework** covers the fallback case where an app hasn't implemented AppFunctions — the agent navigates the UI directly. Both features are in early beta; the post emphasises that privacy and security are first-class design constraints. The framing shift is significant: success is no longer "user opens your app" but "agent successfully fulfils the user's task via your app."

### [Qodana for Android](https://blog.jetbrains.com/qodana/2026/01/qodana-for-android-kotlin/)

Qodana is JetBrains' CI-integrated code quality platform, and this post makes the case for using it alongside (not instead of) Android Lint. While Lint focuses on platform issues (layout validation, resource management, device compatibility), Qodana focuses on Kotlin code quality: null safety misuse, oversized functions, blurred architectural boundaries, and cross-team style divergence. It runs as a CI step, enforcing inspection thresholds and blocking PRs that introduce new violations. The practical case: Lint doesn't catch "this function has 400 lines" or "this module is importing from a module it shouldn't"; Qodana does.
