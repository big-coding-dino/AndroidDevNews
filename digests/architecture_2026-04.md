### [Kotlin 2.3.20 Released](https://blog.jetbrains.com/kotlin/2026/03/kotlin-2-3-20-released/)
Kotlin **2.3.20** is a minor release shipping several useful improvements across the toolchain. On the **Gradle** side, it adds compatibility with **Gradle 9.3.0** and switches Kotlin/JVM compilation to use the **Build Tools API** by default — a step toward cleaner separation between the compiler and build system. **Maven** users get a simplified project setup. **Compiler plugin** updates: the **Lombok plugin** moves to Alpha, and the `kotlin.plugin.jpa` plugin gets improved JPA support (relevant if you're using Hibernate or Spring Data with Kotlin data classes). The **language** gains support for **name-based destructuring declarations**, letting you destructure by property name rather than position — a cleaner alternative to positional destructuring for data classes with many fields. In the **standard library**, there's a new API for creating **immutable copies of `Map.Entry`**, rounding out the immutable collections story. **Kotlin/Native** introduces a new **interoperability mode** for C and Objective-C libraries, which matters for KMP projects targeting iOS or native platforms. Gradle 9.x compatibility and the Build Tools API default are the most immediately impactful changes for Android and KMP teams, since Gradle 9 has been a blocker for some projects. Name-based destructuring is a welcome ergonomic improvement for Kotlin code that works with complex data classes. The Lombok Alpha support is a meaningful step for teams migrating Java codebases that still rely on Lombok annotations.

### [Unidirectional Data Flow as a Functional declaration](https://www.tunjid.com/articles/unidirectional-data-flow-as-a-functional-declaration-6230b74f5d785a7ebc8c2a43)
TJ Dahunsi proposes representing UDF (Unidirectional Data Flow) as a typed function signature: `(Flow<Action>) -> Flow<UiState>`. Rather than reaching for Mobius, MVIKotlin, or Molecule, he builds a lightweight abstraction on plain Kotlin coroutine APIs using two core concepts: a `Mutator<Action, State>` interface and a `Mutation<State>` typealias (`State.() -> State`) that acts as a generic reducer. The key mechanic is `Flow<Action>.toMutationStream()`, which splits a hot action stream into per-type cold `Flow`s via `splitByType()`. Each action subtype gets its own `Flow` and can be independently transformed with whatever operators make sense — `distinctUntilChanged` for simple cases, `mapLatest` to cancel in-flight work (replacing a manual `Job?` reference), or `scan`-based accumulation for nuanced deduplication logic. The per-type flows are then merged back into a `Flow<Mutation<State>>` and reduced via `scan` into a `StateFlow<State>`. Testing benefits directly from this design — because each action-to-mutation pipeline is a cold `Flow` extension, you can feed it a `listOf(...).asFlow()` of actions, call `.reduceInto(initialState)`, and use Turbine to assert each emitted state in sequence — no `StateFlow` timing issues, no `runCurrent()` gymnastics. For teams already comfortable with coroutines, this pattern removes a significant layer of indirection between user intent and state change, though extra object allocations per mutation and solid Flow operator fluency are real tradeoffs to consider.
