### [compose-stability-analyzer](https://github.com/skydoves/compose-stability-analyzer)

A real-time Android Studio / IntelliJ plugin that analyses Jetpack Compose composable stability inline, showing gutter icons, inline hints, and warnings as you type — no build step needed. The `@TraceRecomposition` annotation lets you pinpoint exactly which parameter change triggered a recomposition, and a Gradle task exports stability compatibility reports for diffing across PRs. Configurable: you can enable Strong Skipping mode, add ignored type patterns, change indicator colours to match your theme, and include test source sets in the analysis. Pairs well with Compose compiler metrics reports for a full stability audit.

### [Koog 0.5.0 Is Out: Smarter Tools, Persistent Agents, and Simplified Strategy Design](https://blog.jetbrains.com/ai/2025/10/koog-0-5-0-is-out-smarter-tools-persistent-agents-and-simplified-strategy-design/)

Koog 0.5.0 ships full **Agent2Agent (A2A) protocol support**, making it straightforward to connect multiple Koog agents together into a networked system. The release also introduces a **non-graph strategy API** so you can define and iterate on agent strategies in plain Kotlin without touching the graph DSL — useful for rapid prototyping before committing to a full graph-backed workflow with persistence and checkpointing. Agent persistence and checkpointing received further improvements to reliably save and restore complex multi-step workflows. Start with `AIAgent` and the default strategy, prototype with the non-graph API, then graduate to a graph for production reliability.

### [How Reddit Used the R8 Optimizer for High-Impact Performance Improvements](https://android-developers.googleblog.com/2025/11/how-reddit-used-r8-optimizer-for-high.html)

Reddit's Android team enabled full-mode R8 — tree shaking, method inlining, class merging, and identifier minification — after using the App Performance Score in Play Console to identify improvement potential. The results: measurably faster startup times, fewer slow/frozen frames, reduced ANRs, and a smaller APK size, which also contributed to a higher Play Store rating. R8's class merging and inlining stages are where the biggest performance gains come from, as they eliminate abstraction overhead at the compiled bytecode level. The case study is a strong argument for enabling full R8 mode in production; the companion "Configure and Troubleshoot R8 Keep Rules" post covers how to handle reflection-dependent code that breaks without explicit keep rules.

### [Gemini 3 Is Now Available for AI Assistance in Android Studio](https://android-developers.googleblog.com/2025/11/gemini-3-is-now-available-for-ai.html)

Gemini 3 Pro — Google's new frontier model with significantly improved coding and agentic reasoning — is now available in **Android Studio Otter** (latest stable). The no-cost default tier gets limited Gemini 3 access with a **1 million token context window**; for higher rate limits and longer Agent Mode sessions, add a Gemini API key. Enterprise users on Gemini Code Assist licenses get rolling access starting this week, but IT admins must enable preview models in the Google Cloud console first. Agent Mode is the primary beneficiary: the larger context window and stronger reasoning let it handle multi-file refactors and complex tasks that earlier models would drop mid-way.

### [Configure and Troubleshoot R8 Keep Rules](https://android-developers.googleblog.com/2025/11/configure-and-troubleshoot-r8-keep-rules.html)

R8 is a static analyser — it builds a call graph from direct references and removes everything unreachable. Any code accessed via **reflection, JNI, or dynamic class loading** is invisible to R8 and will be stripped at release unless you add an explicit Keep Rule. A keep rule is an instruction of the form "this class/method/field is a dynamic entry point — do not remove it." The post details how to write targeted rules (prefer `-keepclassmembers` over `-keep` to avoid bloating), how to use `@Keep`, and how to use the R8 retrace tool and `--pg-conf-output` flag to audit what rules are actually in effect. Watch the companion Performance Spotlight Week video on YouTube for a live debugging walkthrough.

### [Leveling Guide for Your Performance Journey](https://android-developers.googleblog.com/2025/11/leveling-guide-for-your-performance.html)

Google's five-level performance maturity framework gives Android teams a concrete roadmap from "we have never thought about performance" to "we run a custom bespoke performance CI pipeline." Level 1 is simply turning on Android Vitals in Play Console (zero effort, automatic field data). Level 2 adds Baseline Profiles. Levels 3–5 introduce local benchmarking, custom Macrobenchmark suites, and full continuous performance tracking. The guide is designed to meet teams where they are — a solo developer can start at level 1 today, while a team of ten might jump straight to level 3. The framing is explicitly against over-investing: match the level to your current resourcing.

### [Growing Kotlin Adoption in Your Company](https://blog.jetbrains.com/kotlin/2025/11/growing-kotlin-adoption-in-your-company-2/)

Part 3 of the "Ultimate Guide to Adopting Kotlin in a Java-Dominated Environment" series, focusing on the expansion phase after your core team is convinced. The key insight is that winning over skeptical Java developers requires both hard factors (showing real code improvements: null safety, concise data classes, sealed hierarchies) and soft factors (a dedicated Slack channel, internal study groups, patience). The post recommends against mandating Kotlin top-down — letting developers see the difference in PRs is more effective than any policy. Companion posts in the series cover getting started, evaluating in real projects, helping decision-makers say yes, and scaling org-wide.

### [Does AI Generate Accessible Android Apps?](https://eevis.codes/blog/2025-11-25/does-ai-generate-accessible-android-apps/)

After six months testing Gemini, Junie, Cursor, and Claude on the same app generation prompt, the conclusion is clear: **none of the tools reliably produce accessible UIs**. Every tool added redundant content descriptions — e.g., a button labeled "Add yarn" also getting a content description "Add new yarn", causing screen readers to announce both. Claude additionally appended action hints ("Tap to...") that screen readers read on top of the label. Touch target sizes, focus order, and missing semantic roles were also consistent failures across all tools. The practical takeaway: AI-generated UI code must be reviewed for accessibility by a human developer; do not ship AI-generated Compose screens without running TalkBack and checking the accessibility tree.
