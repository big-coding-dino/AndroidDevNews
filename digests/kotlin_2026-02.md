### [Join the Kotlin Ecosystem Mentorship Program](https://blog.jetbrains.com/kotlin/2026/02/kotlin-mentorship-program/)

The Kotlin Foundation is launching a mentorship program pairing experienced open-source maintainers with first-time contributors. The goal is helping developers make their first meaningful contribution to Kotlin ecosystem libraries, with JetBrains swag for participants and a KotlinConf trip for standout contributors. ### [Compose Multiplatform for Web Goes Beta](https://blog.jetbrains.com/kotlin/2026/02/compose-multiplatform-web-beta/)

CMP 1.9.0 promotes the web (Wasm) target from Alpha to Beta, signaling it's ready for early-adopter production use. The Beta label means the API is stabilizing and the team is committing to migration guides for breaking changes. Web support uses Kotlin/Wasm compiled to WebAssembly, running Compose UI in the browser with no JavaScript framework dependency. Teams targeting Android + iOS + Desktop can now add web as a fourth target with the same shared UI code. Known limitations: DOM interop and SEO remain restricted by the Wasm execution model.

### [Do I Need an Umbrella Framework for My Kotlin Multiplatform App?](https://medium.com/@pamela/kmp-umbrella-framework)

For KMP projects targeting Apple platforms with multiple modules, an umbrella framework is required — without it, each module exports a separate `.framework` file, which the iOS app must individually link. An umbrella framework aggregates all KMP modules into a single `.xcframework` for iOS consumption. The short answer: if you're targeting Apple + multi-module, yes you need one. The post explains the Gradle configuration to produce it and when single-module projects can skip this step.

### [How kotlinx.serialization Generates Code: A Compiler Plugin Deep Dive](https://blog.sebastianfraisse.dev/kotlinx-serialization-compiler-plugin/)

`@Serializable` triggers a Kotlin compiler plugin that generates a companion object implementing `KSerializer<T>` — this happens at compile time with no reflection at runtime. The plugin reads the class's constructor parameters and properties to generate `serialize` and `deserialize` implementations specific to that class's shape. Key implication: sealed class hierarchies require each subtype to also be `@Serializable`; the plugin generates a `SealedClassSerializer` that dispatches based on a type discriminator. Understanding the generation model explains why certain patterns (generic classes, non-primary constructors) require extra annotations.

### [Koin-Detekt: 58 Static Analysis Rules for Koin 4.x](https://github.com/androidbroadcast/detekt-koin4-rules)

A Detekt extension (`dev.androidbroadcast.rules.koin:detekt-koin4-rules:1.1.0`) with 58 rules across 6 categories: Service Locator anti-patterns, Module DSL correctness, Scope Management, Platform-specific usage, Architecture violations, and Koin Annotations validation. Catches common Koin mistakes at edit time — missing module declarations, incorrect qualifiers, scope mismatches — before runtime. Add to `.detekt.yml` to enable individual rules; integrates with existing Detekt CI pipelines.

### [Koog × ACP: Connect an Agent to Your IDE](https://blog.jetbrains.com/ai/2026/02/koog-acp-ide-integration/)

The Agent Communication Protocol (ACP) enables Koog agents to connect to JetBrains IDEs and other ACP-compliant tools. This post covers the missing piece from earlier tutorials: how to make a Koog agent ACP-compliant using the `acp` module. Once ACP-compliant, the agent exposes a standard interface that IDEs, CI systems, and other agents can communicate with — enabling use cases like an agent that can read your open file, run inspections, and apply fixes directly in the editor. Requires Koog 0.5+ and a JetBrains IDE with the AI plugin.

### [Re: Dependency Injection vs. Service Locators](https://medium.com/@developer/di-vs-service-locator-kotlin)

A clear-eyed taxonomy of the Dagger/Hilt/Metro vs. Koin/Kodein debate: the former are true DI containers (dependencies declared at construction time, graph validated at compile time), the latter are service locators (dependencies retrieved at use time, graph validated at runtime). The practical tradeoffs: DI catches missing/misconfigured bindings at compile time but has higher setup overhead; service locators are simpler to configure but fail at runtime. For KMP projects, service locators have better multiplatform support today. The post argues the choice should be driven by team size, test strategy, and compile-time safety requirements — not framework popularity.

### [Jetpack Compose and the Speed of Thinking](https://dev.to/developer/compose-speed-of-thinking)

A practitioner's account of the Compose learning curve and what clicked — the shift from "layouts as XML trees I modify imperatively" to "UI as a function of state I declare." Once that mental model lands, features like `remember`, `derivedStateOf`, and recomposition become intuitive rather than mysterious. The post is honest about the initially steep learning curve with `mutableStateOf` terminology and recommends approaching Compose by building small throwaway UIs before tackling production screens. Useful for teams still on the fence about migrating from Views.

### [Java to Kotlin Conversion Comes to Visual Studio Code](https://blog.jetbrains.com/kotlin/2026/02/j2k-vscode/)

JetBrains' Java-to-Kotlin (J2K) converter, previously IntelliJ IDEA-only, is now available in the Kotlin VS Code extension. Select a `.java` file and invoke "Convert Java to Kotlin" — the converter produces idiomatic Kotlin with null safety, data classes where appropriate, and Kotlin-style property access. The conversion is heuristic-based so output still needs review, but it eliminates the mechanical rewrite work. Significant for Android developers who use VS Code for non-Android Kotlin work (backend, scripts, KMP shared modules).

### [Qodana for Android: Increasing Code Quality for Kotlin-First Teams](https://blog.jetbrains.com/qodana/2026/02/qodana-android/)

Qodana's Android profile adds static analysis tailored to Kotlin-first Android codebases: Compose-specific inspections, coroutine misuse detection, resource leak analysis, and Android API level compatibility checks. Runs in CI as a Docker container or GitHub Action; results surface in the JetBrains platform and can gate PRs. For teams already using Detekt, Qodana adds a complementary layer of IDE-backed inspections that Detekt's AST-level rules can't cover (e.g., data flow analysis, cross-file reference checking).
