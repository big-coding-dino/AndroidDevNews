### [Kotlin 2.3.20 Released](https://blog.jetbrains.com/kotlin/2026/03/kotlin-2-3-20-released/)
Kotlin **2.3.20** is a minor release shipping several useful improvements across the toolchain. On the **Gradle** side, it adds compatibility with **Gradle 9.3.0** and switches Kotlin/JVM compilation to use the **Build Tools API** by default — a step toward cleaner separation between the compiler and build system. **Maven** users get a simplified project setup. **Compiler plugin** updates: the **Lombok plugin** moves to Alpha, and the `kotlin.plugin.jpa` plugin gets improved JPA support (relevant if you're using Hibernate or Spring Data with Kotlin data classes). The **language** gains support for **name-based destructuring declarations**, letting you destructure by property name rather than position — a cleaner alternative to positional destructuring for data classes with many fields. In the **standard library**, there's a new API for creating **immutable copies of `Map.Entry`**, rounding out the immutable collections story. **Kotlin/Native** introduces a new **interoperability mode** for C and Objective-C libraries, which matters for KMP projects targeting iOS or native platforms. To update: bump the Kotlin version to `2.3.20` in your build scripts, or update IntelliJ IDEA / Android Studio to its latest version, which bundles the release automatically.

### [An update on Android KTX](https://jakewharton.com/an-update-on-android-ktx/)
After eight years, Jake Wharton announced the wind-down of the **Android KTX** (and all `-ktx`) extension libraries — not because they failed, but because they succeeded. Kotlin adoption across the AndroidX ecosystem has been thorough enough that all KTX extensions have been merged directly into their respective main libraries, making the `-ktx` modules empty and obsolete. The KTX story started with `core-ktx` (originally "Android KTX") co-authored by Wharton, Chris Banes, and Romain Guy. Over time it expanded into dozens of libraries — `fragment-ktx`, `lifecycle-viewmodel-ktx`, `work-runtime-ktx`, and more — as a way to bolt Kotlin-friendly APIs onto Java-first AndroidX libraries. The end goal was always to make those main libraries Kotlin-first themselves, which has now happened. Notable deprecation milestones include `activity-ktx` (obsolete in 1.9.0), `core-ktx` (obsolete in 1.19.0-alpha01), `fragment-ktx` (obsolete in 1.9.0-alpha01), and `security-crypto-ktx` which has no migration path — the entire `security-crypto` library is deprecated and should be removed entirely. A Lint feature request is in progress to warn when you declare a `-ktx` dependency at or above its obsolescence version, which would help automate migration.

### [RikkaIcons](https://github.com/rainxchzed/RikkaIcons)
**RikkaIcons** (`dev.rikkaui.icons`) is a semantic icon token system for Compose Multiplatform that decouples icon identity from icon rendering. Instead of importing specific vectors or glyphs, you reference stable tokens like `RikkaIcons.Heart` — the active `IconPack` handles whether that renders as a font glyph, SVG `ImageVector`, outline, filled, or rounded variant. Swapping your entire icon set is a one-line change at `ProvideIconPack`. The library ships composable modules: the required core pair gives you `IconToken`, `IconPack`, `AppIcon`, and `ProvideIconPack`, plus 229 universal tokens. Six icon packs are available as separate artifacts: `pack-lucide` (1,573 icons), `pack-tabler` (5,039 + 1,053 filled), `pack-phosphor` (1,512 × 3 styles), `pack-bootstrap` (1,408 + 670), and three Material Symbols variants (outlined, rounded, sharp — 810 mapped icons each). Material Symbols uses a variable font (~800 KB), replacing thousands of SVG vectors with a single font file that supports `weight`, `grade`, `opticalSize`, and `fill` axes at runtime. Pack variants are straightforward to configure: `ProvideIconPack(TablerPack.Outline) { ... }` or `ProvideIconPack(rememberMaterialSymbolsPack(MaterialSymbolsVariant.Rounded)) { ... }`. Platform support covers Android (API 24+), Desktop JVM, iOS (arm64, simulator arm64, x64), JavaScript, and WasmJs.

### [cw-json](https://commonsware.com/blog/2026/04/07/announcing-cw-json.html)
Mark Allison (CommonsWare) has published **cw-json**, his first Kotlin Multiplatform library, marking a return to open-source library publishing after a long hiatus. The library is a collection of JSON-related artifacts, with the initial release focused on **JSON Pointer** support. Two artifacts are currently available: **`com.commonsware.json:kmp-jsonpointer`** — a pure KMP implementation of the JSON Pointer spec, providing a `JsonPointer` type with functions for creating and manipulating pointers — and **`com.commonsware.json:kmp-jsonpointer-kxs`** — an extension that integrates JSON Pointer with **Kotlin Serialization**, letting you read and mutate `JsonObject` and `JsonArray` values using pointer-based access. Both artifacts are pure Kotlin, platform-agnostic, and target most major KMP platforms. The library is described as a foundation for something larger still in progress.

### [Kotlin Multiplatform @ Google](https://developer.android.com/kotlin/multiplatform)
Google officially supports **Kotlin Multiplatform (KMP)** for sharing business logic between Android and iOS, and declares it stable and production-ready. Combined with JetBrains' **Compose Multiplatform (CMP)**, developers can now share UI code across platforms as well. The **Kotlin Multiplatform Plugin** for the IDE ships with a new project wizard, preflight environment checks, unified run/debug/test configurations for Android and iOS, and basic Swift support including cross-language debugging and navigation. A significant portion of the **Jetpack library ecosystem** has been migrated to KMP, including `annotation` and `compose` (April 2026), `collection` (March 2026), `datastore`, `lifecycle`, `viewModel`, `viewModel-compose`, `navigation`, `savedstate` (March 2026), `navigation3`, `navigationevent`, `paging`, `sqlite` (April 2026), and `room` (November 2025). Platform tiers define support levels: **Tier 1** covers Android, JVM, and iOS; **Tier 2** adds macOS and Linux; **Tier 3** includes watchOS, tvOS, Windows, JavaScript, and WASM. Tooling support extends to a KMP module wizard, **Live Edit** for Compose Multiplatform targeting `androidMain`, and **Previews** from the `commonMain` source set.

### [Unidirectional Data Flow as a Functional declaration](https://www.tunjid.com/articles/unidirectional-data-flow-as-a-functional-declaration-6230b74f5d785a7ebc8c2a43)
TJ Dahunsi proposes representing UDF (Unidirectional Data Flow) as a typed function signature: `(Flow<Action>) -> Flow<UiState>`. Rather than reaching for Mobius, MVIKotlin, or Molecule, he builds a lightweight abstraction on plain Kotlin coroutine APIs using two core concepts: a `Mutator<Action, State>` interface and a `Mutation<State>` typealias (`State.() -> State`) that acts as a generic reducer. The key mechanic is `Flow<Action>.toMutationStream()`, which splits a hot action stream into per-type cold `Flow`s via `splitByType()`. Each action subtype gets its own `Flow` and can be independently transformed with whatever operators make sense — `distinctUntilChanged` for simple cases, `mapLatest` to cancel in-flight work, or `scan`-based accumulation for nuanced deduplication logic. Testing benefits directly from this design: because each action-to-mutation pipeline is a cold `Flow` extension, you can feed it a `listOf(...).asFlow()` of actions, call `.reduceInto(initialState)`, and use Turbine to assert each emitted state in sequence — no `StateFlow` timing issues, no `runCurrent()` gymnastics.
