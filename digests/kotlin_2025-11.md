### [Understanding the Native SDK Wrapper Pattern in Kotlin Multiplatform](https://www.revenuecat.com/blog/engineering/kmp-wrapper-pattern/)

RevenueCat's Purchases KMP SDK wraps the existing native Android and iOS SDKs rather than reimplementing them from scratch, using a delegation pattern with bidirectional type conversion. The design avoids data duplication by mapping KMP types to native types at the boundary and maintaining state consistency through the native SDK. The key engineering insight: wrapping battle-tested native code is lower risk than rewriting it — you inherit existing test coverage, edge-case handling, and platform-specific optimisations. The article covers the delegation implementation in detail including how the conversion layer is generated.

### [Mastering Kotlin Coroutines: Dispatchers, Jobs, and Structured Concurrency](https://carrion.dev/en/posts/kotlin-coroutines-dispatchers-jobs/)

The authoritative mental model guide for the three coroutine primitives: `Dispatcher` (which thread pool), `Job` (lifecycle handle supporting cancellation and parent/child relations), and structured concurrency (child coroutines inherit the parent scope; cancellation propagates downward; errors don't leak silently). Key rules: use `withContext` to switch dispatchers inside a suspend function; use `launch` for fire-and-forget in a lifecycle scope; never use `GlobalScope` in production. The post includes production-ready patterns for `supervisorScope` (sibling failure isolation) and how `Job` vs `SupervisorJob` affect error propagation.

### [Interfacing with Wasm from Kotlin](https://dev.to/charlietap/interfacing-with-wasm-from-kotlin-2k2c)

You can call WebAssembly modules from Kotlin using Chasm, a Wasm runtime for Kotlin Multiplatform. Chasm provides interface generation for the Wasm module and a runtime that works across JVM, JS, and Native. The practical motivation: Wasm is a compilation target for nearly every language, so Kotlin+Wasm interop is a path to reusing libraries from Python, Rust, Go, Swift, etc. without native bindings. The tooling is early-stage but the direction matters — a future "npm for Wasm" would make this a primary cross-ecosystem integration mechanism.

### [Koin IDE Plugin 1.5.1: Koin Annotations 2.2 and JSR-330 Support](https://blog.kotzilla.io/koin-ide-plugin-1-5-1-annotations-2-2-jsr-330-android-kotlin)

Plugin 1.5.1 adds full understanding of Koin Annotations 2.2, including the new JSR-330 annotation support (`@Inject`, `@Named`, etc.). The plugin now validates annotation-driven DI configurations at edit time — missing module declarations, incorrect qualifiers, and scope mismatches are flagged before you run the app. JSR-330 support is significant for teams migrating from Dagger/Hilt, as it means the same `@Inject` annotations work with Koin's compile-time safety checker.

### [Koog 0.5.0: A2A Protocol, Non-Graph Strategies, Persistent Agents](https://blog.jetbrains.com/ai/2025/10/koog-0-5-0-is-out-smarter-tools-persistent-agents-and-simplified-strategy-design/)

Koog 0.5.0 adds full **Agent2Agent (A2A) protocol** support for interconnecting agents, a **non-graph strategy API** for rapid prototyping without graph DSL, and improved persistence/checkpointing for long-running workflows. The recommended development path: start with `AIAgent` and default strategy → prototype with non-graph API → graduate to graph-based strategy for production reliability. The A2A protocol means Koog agents can communicate with agents built in other frameworks.

### [Using Navigation 3 with Compose Multiplatform](https://johnoreilly.dev/posts/navigation3-cmp/)

Navigation 3 is available for Compose Multiplatform via a JetBrains-maintained fork. All navigation code lives in `commonMain` — the back stack is a plain Kotlin list, adding/removing items triggers the `NavDisplay` to update. The article uses JetBrains' FantasyPremierLeague sample and shows how to add `material3-adaptive-navigation3` for two-pane adaptive layouts with the same API. Dependencies: use the JetBrains versions (`androidxNavigation3UI = "1.0.0-alpha04"`) rather than the Google ones when targeting non-Android platforms.

### [Jetpack Navigation 3 Is Stable](https://android-developers.googleblog.com/2025/11/jetpack-navigation-3-is-stable.html)

Navigation 3 v1.0 reaches stable. The library is built from scratch to embrace Compose state — `NavDisplay` observes a list of keys backed by `Compose state` and re-renders when it changes. This replaces the Nav2 pattern of imperative navigation commands on a `NavController`. The back stack is fully controllable (add/remove any key, not just push/pop), state retention is explicit, and adaptive list-detail layouts are first-class. JetBrains is already using Nav3 in the KotlinConf app. A CMP-compatible version is also available.

### [LitmusKt: A Concurrency Testing Tool That Improved the Kotlin Compiler](https://blog.jetbrains.com/research/2025/10/litmuskt-concurrency-testing/)

JetBrains Research built LitmusKt to systematically expose platform-specific concurrency bugs in Kotlin's JVM, Native, and JavaScript backends. Traditional testing misses these because they're non-deterministic — LitmusKt uses litmus test patterns from memory model research to exhaustively probe the happens-before relationships the runtime guarantees. Running LitmusKt on the Kotlin compiler's own test suite found and fixed real bugs. Relevant for library authors writing concurrent code targeting multiple Kotlin backends.

### [Amper 0.8.0: Compose Hot Reload, JS/Wasm Targets, Gradle Integration Removed](https://blog.jetbrains.com/amper/2025/10/amper-update-october-2025/)

Amper 0.8.0 ships **Compose Hot Reload** support — enabled by default in 0.8.1, just click "Run with Compose Hot Reload" in IntelliJ IDEA for any JVM desktop module. New platform targets: JavaScript and Wasm. The Gradle integration is removed in favour of Amper's own build model. Requires IntelliJ IDEA 2025.3 EAP or newer.

### [KToon: Efficient Serialization for KMP](https://github.com/JosephSanjaya/ktoon)

KToon implements TOON (Token-Oriented Object Notation), a compact binary-like text format that achieves 30–60% smaller payloads than JSON by using column-header style encoding for lists of objects. Drop-in replacement for kotlinx.serialization — annotate with `@Serializable`, switch the format. Targets Android, iOS, Desktop, JS, WASM. Built during a hackathon; treat as experimental.

### [Seamless KMP on iOS: SKIE for Swift-Native APIs](https://carrion.dev/en/posts/kmp-ios-skie-integration/)

SKIE (Swift Kotlin Interface Enhancer, by Touchlab) transforms the Objective-C-based KMP iOS framework into a Swift-native API surface: sealed classes become Swift `enum` with exhaustive `switch`, suspend functions become `async/await`, and `Flow` becomes `AsyncSequence`. Without SKIE, iOS developers consuming KMP code face sealed classes as open classes, coroutines as callback hell, and no native Flow support. SKIE is the pragmatic bridge until JetBrains' `swift-export` feature matures into stable.
