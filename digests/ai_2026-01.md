### [Building AI Agents in Kotlin – Part 4: Delegation and Sub-Agents](https://blog.jetbrains.com/ai/2026/01/building-ai-agents-in-kotlin-part-4-delegation-and-sub-agents/)

Part 4 of the JetBrains Koog series introduces **sub-agents** as a cost and performance optimisation — specifically a dedicated `find` sub-agent that handles file-search tasks before passing results to the main coding agent. The core insight is context growth: every LLM request includes the full conversation history, so every subsequent call is more expensive than the last. By delegating bounded tasks (like "find all files matching this pattern") to a focused sub-agent with a smaller context, you reduce total input tokens and often improve accuracy. Koog's sub-agent API wraps this pattern cleanly: create a child agent, invoke it from a tool in the parent agent, and return a structured result. The hypothesis — lower cost, same or better performance — is validated by measurement in the post.

### [LLM Flexibility, Agent Mode Improvements in Android Studio Otter 3 Feature Drop](https://android-developers.googleblog.com/2026/01/llm-flexibility-agent-mode-improvements.html)

Android Studio Otter 3 is a major agentic upgrade. **Bring Your Own Model (BYOM)** lets you connect any LLM to Android Studio's AI features — not just Gemini. **Agent Mode** now supports device interaction (running your app on an emulator and reacting to what it sees), multiple conversation threads, and **MCP (Model Context Protocol)** server connections so agents can call external tools. **Journeys** lets you write user journey tests in natural language that Agent Mode executes. The new App Links Assistant generates and validates deep links via natural language description. **Automatic Logcat retracing** deobfuscates R8-minified stack traces in real time. This release represents the most significant expansion of Android Studio's agentic surface to date.

### [The AI Coding Ladder – Fragmented Ep. 301](https://fragmentedpodcast.com/episodes/301/)

Kaushik Gopal and Iury Souza map "AI coding" into four distinct paradigms, each requiring a different mental model: **Gen 1** — super autocomplete (Copilot/Sweep style); **Gen 2** — Chat Oriented Programming (CHOP), where you describe changes in a chat and apply diffs; **Gen 3** — Agent (Claude Code, Cursor Agent, Junie), where the AI has tools and acts autonomously in a ReAct loop; **Gen 4** — Agent Orchestration, where multiple specialised agents collaborate on a task. The episode argues that most developers are still using Gen 1/2 tooling with Gen 3/4 tools, leading to frustration. Practical tips: use compact agent-handoff commands, ask the agent to clarify requirements before executing, and use TUI harnesses (Claude Code, Aider) for headless agentic work.

### [How 16 AI Agents Fixed Our Documentation Problem](https://jonnyzzz.com/blog/2026/01/24/16-ai-agents-documentation-refactor/)

A real-world multi-agent orchestration case study: 2,648 lines of documentation across four files were refactored using 16 parallel AI agents, reducing total length by 39% and promoting the most-important quick-start command from line 91 to the top. The breakthrough was using agents to *interview other agents* about what was confusing — the docs were about multi-agent orchestration, for agents spawning other agents, so the feedback was structurally recursive. The key engineering lesson: stop guessing what's confusing and ask. The multi-agent approach enabled parallelism (different agents working different files simultaneously) and specialisation (one agent type for interviewing, another for rewriting, another for validation). Two prior attempts using a single agent had failed.

### [Building AI Agents in Kotlin – Part 5: Teaching Agents to Forget](https://blog.jetbrains.com/ai/2026/01/building-ai-agents-in-kotlin-part-5-teaching-agents-to-forget/)

Part 5 tackles the inevitable context-window exhaustion problem: agents that run long tasks accumulate enormous context (every file read, every command output, every search result) and eventually crash or degrade. The solution is **threshold-based history compression** — once context exceeds a configurable token threshold, a compression step summarises the least-recently-useful history down to key decisions and current state. The post notes the series switched from GPT-5 Codex (0.58 SWE-bench Verified) to Claude Sonnet 4.5 (0.60, faster) mid-series, but Claude's 200K context window still fails on sufficiently complex tasks. The core principle: you cannot buy enough context; agents must learn to forget strategically, retaining only what they need to complete the current step.
