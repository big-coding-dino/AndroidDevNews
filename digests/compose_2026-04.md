### [RikkaIcons](https://github.com/rainxchzed/RikkaIcons)
RikkaIcons (`dev.rikkaui.icons`) is a semantic icon token system for Compose Multiplatform that decouples icon identity from icon rendering. Instead of importing specific vectors or glyphs, you reference stable tokens like `RikkaIcons.Heart` — the active `IconPack` handles whether that renders as a font glyph, SVG `ImageVector`, outline, filled, or rounded variant. Swapping your entire icon set is a one-line change at `ProvideIconPack`. The library ships with core modules giving you `IconToken`, `IconPack`, `AppIcon`, and `ProvideIconPack`, plus 229 universal tokens and optional extended layers adding hundreds more. Six icon packs are available as separate artifacts, including `pack-tabler` (5,039 + 1,053 filled), `pack-phosphor` (1,512 × 3 styles), and Material Symbols variants using a variable font (~800 KB) instead of hundreds of individual `ImageVector` objects. Platform support covers Android, Desktop JVM, iOS, JavaScript, and WasmJs.

### [Unidirectional Data Flow as a Functional declaration](https://www.tunjid.com/articles/unidirectional-data-flow-as-a-functional-declaration-6230b74f5d785a7ebc8c2a43)
TJ Dahunsi proposes representing UDF as a typed function signature: `(Flow<Action>) -> Flow<UiState>`, building a lightweight abstraction on plain Kotlin coroutine APIs using a `Mutator<Action, State>` interface and a `Mutation<State>` typealias (`State.() -> State`) as a generic reducer. The key mechanic is `Flow<Action>.toMutationStream()`, which splits a hot action stream into per-type cold `Flow`s via `splitByType()`, each independently transformable with operators like `distinctUntilChanged`, `mapLatest`, or `scan`-based accumulation before merging back into a `Flow<Mutation<State>>` and reducing via `scan` into a `StateFlow<State>`. Testing benefits directly from this design — each action-to-mutation pipeline is a cold `Flow` extension, so you can feed it a `listOf(...).asFlow()` of actions, call `.reduceInto(initialState)`, and use Turbine to assert each emitted state in sequence without `StateFlow` timing issues. For teams already comfortable with coroutines, this pattern removes a significant layer of indirection between user intent and state change.

### [Shared element transitions for large screened devices](https://www.tunjid.com/articles/shared-element-transitions-for-large-screened-devices-6936d332566f1145a11726a8)
Shared element transitions in Jetpack Compose work cleanly on phones but break on large screens — tablets, foldables, desktop — where multiple navigation entries stay visible simultaneously. The root issue is that Compose's shared element APIs only support a "rule of two" (one invisible source, one visible target), which breaks when a List→Detail layout keeps the list pane on screen after navigation, causing elements to disappear or leave visual holes. The fix is a "Composable Sandwich" pattern that decouples bounds tracking from rendering logic: wrap content in a parent `Box`, nest a tracker `Box` with the `sharedElement` modifier for animation math only, and conditionally place content as a child of the tracker (when animating) or as a sibling (when just visible). For three-or-more-elements cases, you assign roles — source, target, and "others" — where others pass a static `AnimatedVisibilityScope` via `rememberStaticExitedAnimatedVisibilityScope()` to permanently hold `targetState = EnterExitState.PostExit`. This pattern gives you a concrete path to preserve shared element polish across all form factors without waiting for an API update.

### [Migrating a Jetpack Compose app to Kotlin Multiplatform](https://kotlinlang.org/docs/multiplatform/migrate-from-android.html)
This guide walks through migrating Jetcaster, a production-ready Android podcast app built with Jetpack Compose, to Kotlin Multiplatform (KMP) so it runs on Android, iOS, and desktop — demonstrated as a commit-by-commit follow-along on a public repository. The key preparation steps are converting Java code to Kotlin (since `commonMain` cannot contain Java), replacing Android-only dependencies with multiplatform equivalents (Dagger/Hilt → Koin, Coil 2 → Coil 3, ROME RSS parser → multiplatform RSS Parser, JUnit → kotlin-test), and isolating any Android Views code in `androidMain` or rewriting in Compose. The migration follows a phased sequence: migrate multiplatform libraries first, rewrite Java-dependent code into Kotlin, migrate business logic modules (`:core:data`, `:core:domain`) with Room v2.7.0+ and `kotlinx-datetime`, then migrate UI screens one at a time. New platform entry points are added last — a JVM/desktop `main()` function enables Compose Hot Reload, and iOS connects via Swift UI to the shared `JetcasterApp` composable. The guide honestly addresses friction points: `java.time` rewrites, Android-only libraries like Hilt, and resource handling differences between Android (`res`, `R`) and multiplatform Compose (`composeResources`, `Res`).
