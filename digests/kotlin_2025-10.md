### [Kotlin Mutex: Thread-Safe Concurrency for Coroutines](https://carrion.dev/en/posts/kotlin-mutex-concurrency-guide/)

`Mutex` provides coroutine-friendly mutual exclusion without blocking threads — use `mutex.withLock { }` for automatic acquisition and release. Key rules: prefer `Mutex` over `synchronized` in coroutine code; for simple counters, use `AtomicInteger` instead; for multiple concurrent access slots, use `Semaphore`. The article covers how `Mutex` interacts with coroutine cancellation (the lock is released on cancellation) and when `Actor` or `StateFlow` are better fits for complex state management. A practical reference for anyone hitting race conditions in `viewModelScope` or background coroutines.

### [Understanding the Internals of Flow, StateFlow, and SharedFlow](https://www.revenuecat.com/blog/engineering/flow-internals/)

A deep dive into the machinery underneath Kotlin's reactive stream abstractions. The two core properties of `Flow` — context preservation (a flow carries its own coroutine context) and exception transparency — explain why you can't call non-suspend code inside a flow operator and expect it to behave predictably. `SharedFlow` uses a circular buffer with configurable replay and extra-buffer capacity; `StateFlow` is a specialisation with replay=1 and conflation. Understanding the buffering behaviour is essential for reasoning about backpressure and why `buffer()` and `conflate()` operators exist.

### [Zero-Cost Abstractions in Kotlin: Inline Functions and Value Classes](https://carrion.dev/en/posts/kotlin-inline-functions-value-classes/)

`inline` functions eliminate call-site overhead for small higher-order utilities and enable `reified` type parameters — use them for frequently called lambdas and type-safe generic helpers. `value class` wraps a single value with strong type safety and zero allocation at runtime in most cases. The key gotchas: value classes get boxed when used with generics or nullable types, losing the performance benefit; inline functions with large bodies bloat call sites. Practical use cases: `value class UserId(val id: String)` prevents mix-ups at compile time with no runtime cost on the happy path.

### [Value Classes Are the New Data Classes](https://curiouslab.dev/0002-value-classes-are-new-data-casses.html)

With Project Valhalla publishing an early-access JDK build for value classes, this post examines Kotlin's own value class plans in that context. Currently, Kotlin value classes use boxing as a fallback for generics and interface implementation — a limitation that Valhalla's primitive objects will eventually eliminate for JVM targets. The post traces the theoretical basis (product types vs. value types) and argues that `value class` is a better model than `data class` for most domain value objects that don't need identity semantics.
