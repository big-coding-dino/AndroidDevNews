### [Explore AI on Android with the Sample Catalog App](https://android-developers.googleblog.com/2025/12/explore-ai-on-android-with-our-sample.html)

Google launched a redesigned **Android AI Sample Catalog** — an open-source app that bundles runnable demos of every major on-device and cloud AI API in a single place. ### [Building AI Agents in Kotlin – Part 1: A Minimal Coding Agent](https://blog.jetbrains.com/ai/2025/11/building-ai-agents-in-kotlin-part-1-a-minimal-coding-agent/)

The first installment of JetBrains' hands-on Koog agent-building series starts with three tools — list directory, read file, and edit file — and builds a minimal coding agent in Kotlin. The framing is crucial: you're not writing code that does things, you're giving an LLM the ability to do things and letting it decide when. The post focuses on the tool selection problem (too many tools → poor choices; too few → can't complete the task) and how to observe agent behaviour to find the right balance. Later parts in the series extend this with shell execution, tracing, sub-agents, and history compression.

### [Building AI Agents in Kotlin – Part 2: A Deeper Dive Into Tools](https://blog.jetbrains.com/ai/2025/11/building-ai-agents-in-kotlin-part-2-a-deeper-dive-into-tools/)

Part 2 adds an `ExecuteShellCommandTool` to the coding agent from Part 1, closing the core engineering feedback loop: write code → compile → observe failures → fix. The post explains that while LLMs handle syntax well, they commonly miss nonexistent method calls, forgotten imports, and partial interface implementations — errors that only surface at compile time. The anatomy of a Koog tool requires implementing a specific abstract class and returning structured output; the shell tool wraps `ProcessBuilder` and captures stdout/stderr. The key design lesson: a small amount of prompting ("run a quick test to validate your output before finishing") dramatically reduces the number of LLM turns needed to reach a working result.

### [YARC – Yet Another Rapid Compose](https://github.com/andrew-malitchuk/yet-another-rapid-compose)

An Android Studio plugin inspired by Emmet that expands shorthand abbreviations into full Jetpack Compose component trees. Type `col/row/txt` and expand to a `Column` containing a `Row` containing a `Text`. Supports nesting (`/`), repetition (`*N`), siblings (`,`), and stepping up the hierarchy (`;`). Available via the JetBrains Marketplace. Best suited for quickly scaffolding layout hierarchies during prototyping; generated code is standard Compose so no runtime dependency is introduced.

### [ComposeGuard](https://github.com/AndroidPoet/compose-guard)

A JetBrains/Android Studio plugin that enforces Compose best practices in real time — no build step, no waiting for lint. Implements rules across 7 categories: naming (PascalCase for composables, camelCase for lambdas), modifiers (required modifier param, modifier ordering, avoid `Modifier.composed`), state (remember, mutableStateOf types, state hoisting), parameters (ordering, trailing lambdas, no mutable types as params), composable structure, preview, and more. Each violation shows a gutter icon with a quick fix and detailed explanation. A good complement to the official Compose compiler stability analysis.

### [18% Faster Compiles, 0% Compromises](https://android-developers.googleblog.com/2025/12/18-faster-compiles-0-compromises.html)

The Android Runtime (ART) team reduced compile time by 18% in 2025 with no regressions in compiled code quality or peak memory usage. The improvements cover both JIT (reducing latency before optimisations kick in, leading to a smoother cold-start experience) and AOT compilation (lower resource consumption, better battery and thermals on low-end devices). Some improvements shipped in the June 2025 Android release; the remainder arrive in the year-end release. All users on Android 12+ are eligible to receive these improvements via Mainline module updates — no OS upgrade required.

### [Goodbye Mobile Only, Hello Adaptive: Three Essential Updates from 2025](https://android-developers.googleblog.com/2025/12/goodbye-mobile-only-hello-adaptive.html)

Android 16 removes the ability for apps to lock orientation or resizability on displays ≥600dp — the system ignores manifest and runtime restrictions and fills the window. **WindowManager 1.5.0** introduces large (600–839dp) and extra-large (840dp+) window size classes to replace the old breakpoints. **Jetpack Navigation 3** rethinks navigation with a multi-pane first API, making two-pane and three-pane layouts a first-class pattern rather than a bolt-on. The business case is stark: tablet users spend 9× more on apps than phone-only users; foldable users spend ~14×.

### [How Backend Development Teams Use Kotlin in 2025](https://blog.jetbrains.com/kotlin/2025/12/how-backend-development-teams-use-kotlin-in-2025/)

JetBrains-certified Kotlin trainer José Luis González identifies the top anti-patterns in self-taught backend Kotlin teams: deep inheritance hierarchies carried over from Java (fix: use sealed classes and data-oriented design), nullable types used everywhere as a Java null substitute (fix: use `?` only at system boundaries, not as a general default), and coroutine scope mismanagement including swallowed `CancellationException` in `catch` blocks (fix: always rethrow or use `runCatching` carefully). The practical takeaways are immediately applicable in code review: flag `open class` without a sealed hierarchy, flag bare `catch(e: Exception)` blocks in coroutine contexts.

### [How Mobile Development Teams Use Kotlin in 2025](https://blog.jetbrains.com/kotlin/2025/12/how-mobile-development-teams-use-kotlin-in-2025/)

The companion mobile-focused post from the same author zeroes in on the most common production bug in mobile Kotlin: **swallowing `CancellationException`** inside a general `catch` block or `runCatching`, which disables cooperative coroutine cancellation and causes mysterious timeouts and lifecycle leaks. The fix is always to rethrow `CancellationException`, use `coroutineContext.ensureActive()` in long-running loops, and let cancellation propagate naturally. The post also covers using structured concurrency with `supervisorScope` for parallel tasks where one failure shouldn't cancel siblings, and recommends the detekt `SwallowedException` rule to catch this automatically.

### [The Ultimate Guide to Successfully Adopting Kotlin in a Java-Dominated Environment](https://blog.jetbrains.com/kotlin/2025/12/the-ultimate-guide-to-successfully-adopting-kotlin-in-a-java-dominated-environment/)

The five stages are: getting started (small, safe experiments), evaluating in real projects, growing adoption (winning hearts and minds), helping decision-makers say yes, and scaling organisation-wide. ### [Márton Braun: Hot Reload in Compose and Kotlin New Features (Kotlin Khronicles, Ep. 7)](https://soundcloud.com/kotlin-khronicles/marton-braun-hot-reload-in-compose-and-kotlin-new-features)
