### [OpenPAD: On-Device Face Liveness Detection for Android](https://github.com/iamjosephmj/OpenPAD)

OpenPAD is an open-source Android library for **Presentation Attack Detection** — detecting face spoofing attempts (photos, screens, video replays, face swaps) using an on-device multi-layer ML pipeline. All processing runs locally: no server calls, no data leaves the device. Requires Android 8.0+ (API 26) and a front-facing camera. Add via JitPack (`com.github.iamjosephmj:OpenPAD:1.0.6`), call `OpenPad.initialize(context)` once in `onCreate()`, and use `OpenPadConfig.Banking` for a preset configuration or set `livenessThreshold` / `faceMatchThreshold` directly. Useful for any app with KYC or identity verification requirements.

### [Supercharge Your Android Development: 6 Expert Tips for Gemini in Android Studio](http://android-developers.googleblog.com/2026/03/supercharge-your-android-development.html)

Google and GDEs compiled best practices for using Gemini in Android Studio Otter 3, focusing on the two highest-leverage features: **Agent Mode** and the **New Project Assistant**. The New Project Assistant (in Canary) accepts prompts and optional design mockup images to generate a full app scaffold — architecture, dependencies, Compose layouts — then deploys to the emulator and validates that rendered screens match the spec. Agent Mode tips include: keep conversations focused on one task at a time, explicitly review suggested changes before accepting, and use multiple conversation threads to isolate independent workstreams. The post is the most practical "how to actually use this" guide published since Otter 3 launched.

### [Go from Prompt to Working Prototype with Android Studio Panda 2](https://android-developers.googleblog.com/2026/05/go-from-prompt-to-working-prototype.html)

Android Studio **Panda 2** ships the AI-powered New Project flow as stable. Describe what you want to build (text + optional images), the agent creates a project plan for your review, and once confirmed it runs an autonomous loop: generate code → build → analyse errors → self-correct → repeat until the project compiles and renders. Output uses Kotlin, Compose, and the latest stable Jetpack libraries. The self-correction loop is the key advance over prior "generate and dump" approaches — the agent sees real build output, not just static analysis. Think of it as a one-shot prototype scaffolder where you review the plan once before the agent executes.

### [Elevating AI-Assisted Android Development and Improving LLMs with Android Bench](https://android-developers.googleblog.com/2026/03/elevating-ai-assisted-androi.html)

Google published **Android Bench** — an official public leaderboard that scores LLMs on real-world Android development tasks. The benchmark covers common Android development areas of varying difficulty, curated from actual developer challenges rather than synthetic puzzles. The goal is to give model creators a clear target: here is what high-quality Android development looks like, here is where your model falls short. For developers, the leaderboard answers "which model should I use for Android tasks?" with data instead of vibes. The benchmark is open so third-party models can submit results.

### [Android ❤️ Gemini CLI: Tech Debt Detection with a Code Maat Skill](https://www.maiatoday.net/p/android-%EF%B8%8F-gemini-cli-tech-debt-detection-code-maat-skill/)

A practical walkthrough of building a **Gemini CLI skill** that wraps Code Maat — the forensic git-history analysis tool from Adam Tornhill's "Your Code as a Crime Scene" — to automatically identify hotspots (files changed most frequently), logical coupling (files that change together), and churn in an Android codebase. The skill is loaded lazily by Gemini CLI only when needed, keeping the agent's context clean. The post uses the Habitica Android app (first commit 2015) as a real test case. The key insight: rather than asking the AI to intuit where tech debt lives, you teach it to run a deterministic script and reason about the output. The post also compares this Gemini CLI approach to Android Studio's built-in Gemini agent.

### [Refuelling Your Jetpack: Jetpack APIs on Kotlin Multiplatform](https://dev.to/tkuenneth/refuelling-your-jetpack-d8i)

A mapping of the familiar Jetpack library stack onto Kotlin Multiplatform equivalents, framed as the evolution from Android-only to cross-platform development. The key insight: Jetpack's original purpose was to unbundle features from the OS and provide backward compatibility and guidance — KMP plays an analogous role across platforms. The post walks through which Jetpack libraries have direct KMP counterparts, which have community equivalents, and which require platform-specific implementations. Targeted at Android developers evaluating KMP for the first time who want to understand what they'd gain and lose from their existing library knowledge.

### [Boosting Android Performance: Introducing AutoFDO for the Kernel](https://android-developers.googleblog.com/2026/03/BoostingAndroid%20PerformanceIntroducingAutoFDO.html)

The Android LLVM team is applying **Automatic Feedback-Directed Optimization (AutoFDO)** to the Android kernel — using real-world CPU branch-history profiles from fleet devices to guide the compiler's inlining and branch-prediction decisions, rather than relying on static heuristics. The result is a kernel tuned to actual usage patterns rather than theoretical ones. AutoFDO was already used in userspace Android; extending it to the kernel delivers system-wide improvements that benefit all apps without any app-level changes. The technique requires capturing hardware performance counter data during representative workloads, which is why it's only practical at Google's scale for the kernel itself.

### [Expanding Google Play's Stage for PC and Paid Titles](https://android-developers.googleblog.com/2026/03/building-a-bigger-stage.html)

Google Play is adding **cross-platform PC discovery** (find and buy Android/PC games from a single Play listing), **Game Trials** (let players try before buying), and **buy-once-play-anywhere** pricing across Android and PC. The **Level Up program** is the strategic roadmap for games developers prioritising cross-platform optimisations. The announcement is a signal that Play is treating PC as a first-class distribution target alongside Android, not an afterthought. Relevant for game developers considering whether to invest in PC ports. *(Summary-only — policy/business announcement; limited technical depth.)*

### [Enabling R8 Optimization at Scale with AI-Assisted Debugging](https://engineering.grab.com/r8-optimization-at-scale-with-ai-assisted-debugging)

Grab's Android team (9 million lines of code) enabled full R8 optimization across a superapp that was seeing ANRs spiking platform-wide — not in isolated features, but everywhere, strongly correlated with memory pressure. The process of enabling R8 at that scale required fixing hundreds of keep-rule violations, and Grab used AI-assisted debugging to accelerate the identification and remediation of each crash introduced by R8. The post details the workflow: R8 strips a class → app crashes → AI analyses the stack trace and R8 mapping file together → AI proposes the specific keep rule needed. The outcome was significant improvements in app size, startup time, and ANR rate. A concrete production example of the AI-in-the-build-toolchain pattern described elsewhere in theory.
