**Ep. 247 — The Art of App Modularization with Siggi Jonsson**
*Fragmented Podcast · Don Felker & Kaushik Gopal · 66 min · 2023-06-12*

Siggi Jonsson, formerly of Tinder's client platform team, walks through the why, when, and how of Android app modularization — including the wrong paths he helped pave before landing on what actually works.

The core message: most apps don't need modularization. Adding modules introduces real overhead and doesn't solve all problems. The right trigger isn't a team size number or a blog post — it's pain. Specifically, when developers are stepping on each other's work or when build times are visibly degrading productivity. Build time pain is usually the first signal teams notice: making a small change forces a full app compile before you can verify anything. Siggi notes this used to be mitigated by extracting pure JVM modules (no Android framework dependencies) so unit tests could run fast — but the deeper insight is that if the build tools are struggling to resolve your dependency graph, humans are probably struggling to understand it too.

Tinder's modularization journey started with the layered approach: separate modules for data, domain, and UI, following clean architecture principles. This enforced encapsulation and prevented engineers from, say, making network calls from the UI layer. The tradeoff: all feature logic converged into a small number of fat layer modules — they'd traded one monolith for several. The database question crystallized this: with SQLDelight (or Room), do you scatter per-feature schemas or share one? Sharing one database couples features implicitly at the schema level; per-feature databases add runtime overhead. Tinder kept a shared data layer but eventually had to split even that. The evolution landed on per-feature module stacks — each feature owning its own UI, domain, and data modules — while still maintaining some shared infrastructure. The parallel to microservices "joined at the database" is intentional and instructive.

**Why it's worth your time:** If your team is debating whether to modularize, or you've already started and the graph is getting messy, Siggi's firsthand account of building and then untangling Tinder's module structure gives you a realistic map of the traps and how to think about the dependency graph as a people problem as much as a build problem.
