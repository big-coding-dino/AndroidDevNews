**Ep. 240 — Important Kotlin Constructs**
*Fragmented Podcast · Don Felker & Kaushik Gopal · 55 min · 2022-12-15*

Kaushik wrote a blog post cataloguing five Kotlin constructs that are easy to confuse or lose track of as the language evolves — partly because Kotlin lets you combine keywords in ways that significantly change semantics (`data class` vs `data object` vs `value class`), and partly because some constructs spent time as experimental before stabilizing.

The episode digs into `fun interface` first. A functional interface is an interface with exactly one abstract method (SAM — Single Abstract Method), though it can have additional non-abstract methods. Its main purpose is enabling function-passing patterns, historically a workaround for Java's lack of first-class functions. The canonical Android example is `OnClickListener`: you define the interface once, pass it around, and let callers supply the `onClick` implementation. Adding a second abstract method is a compile error. Kotlin then introduced **function types** as a native alternative — the `(Int) -> String` syntax — which treats functions as first-class types directly rather than wrapping them in a named interface. Kaushik and Don both lean toward function types when possible: less ceremony, no need to name the interface or its abstract method, and the function's context usually provides enough clarity. The tradeoff is that named functional interfaces can add explicitness that some teams prefer.

The episode also touches on a broader point about blogging: writing forces deeper understanding than just using a feature. Don recounts a DroidCon NYC talk on the Activity lifecycle that packed a standing-room-only main auditorium — a reminder that fundamentals always have an audience, even for experienced developers.

**Why it's worth your time:** If you've been using `fun interface` and function types interchangeably without knowing why they exist or when to prefer one over the other, this episode gives you the mental model and historical context to make deliberate choices. The explainer on SAM conversion and how desugaring blurred the distinction before Kotlin's function types is particularly useful.
