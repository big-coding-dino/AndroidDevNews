**Ep. 252 — Everyone needs a starter template**
*Fragmented Podcast · Kaushik Gopal · 25 min · 2026-11-26*

Setting up a new Android project is decision paralysis in a box: Compose version, navigation approach, DI framework, JSON library, logging — all before writing a line of actual code. The host's answer is a personal starter template ("Playground Android App") that encodes those decisions upfront. Key choices in his template: Gradle version catalog as the single source of truth (SDK versions, namespace, version code all live there), convention plugins to collapse multi-module boilerplate to one line, Makefiles for fast targeted tasks (run unit tests only, nuke build dirs, kill the Kotlin daemon), curated lint rules borrowed from Square/Slack/Uber, multi-module structure pre-wired, and Kotlin Inject + Anvil for DI across modules.

The second half brings in Martin Brown, developer advocate at JetBrains, to discuss Amper — a new build tool aimed at exactly this problem. Amper comes in two flavors: a Gradle plugin layer (good for incremental adoption, supports Gradle version catalogs) and a standalone build tool that only calls into Gradle when forced (e.g., Android Gradle Plugin). The core design choice is a *declarative* YAML config — no if/for logic, fixed schema — which makes IDE tooling, auto-complete, and automated config edits reliable in ways an imperative Gradle DSL can't match. Tradeoff: you lose the escape hatch of arbitrary scripting, though Amper is selectively opening up versioning knobs (Compose version, KotlinX serialization version) where real demand exists. New project onboarding is a single file with IDE quick-fixes guiding you through setup; a CLI is also available. Amper also aims to handle environment bootstrapping (JDK, Android SDK) automatically, moving toward the deterministic sandbox model that Python/Rust/Ruby ecosystems already have.

**Why it's worth your time:** The starter template walkthrough is a practical checklist you can steal directly — Gradle convention plugins and Makefiles alone will save hours on every new project. The Amper interview is a useful early look at where Android build tooling may be heading, with an honest discussion of the declarative vs. imperative tradeoff that frames why this is a hard problem in the first place.
